üìã Overview

Purpose: Deliver fast account inquiries (Account View) and secure edits (Account Update) by surfacing balances, cards, customer info, and status chips in one responsive layout.

Business Context: Back-office support teams and compliance administrators rely on these screens to validate identity, wait-on account lifecycle decisions, and reconcile disputes without switching applications.

  • 11-digit account lookup with client-side validation + padded query parameters for `/account-view`.
  • Masked SSN/card numbers with a toggle so compliance is maintained until a rep explicitly reveals sensitive values.
  • Update mode guarded by a switch + confirmation dialog before calling `PUT /accounts/{accountId}`.

üéØ User Story Development Guide

üìù Specific Templates

Support lookup

As a back-office representative, I want to search by 11-digit account ID so that I can read balances, active cards, and customer contact data on the same screen.

Account update

As an admin, I want to toggle edit mode, update limits and contact info, and confirm the submission before saving so that the account and customer records stay synchronized.

Compliance review

As a compliance analyst, I want to temporarily reveal SSN/card numbers via a single toggle and then re-mask them so that sensitive data is protected by default.

‚ö° Development Acceleration Factors

  • `useAccountView` + `useAccountUpdate` hooks: Encapsulate API calls, loading/error states, initialization, and change detection so the pages remain focused on rendering.
  • `apiClient`: Automatically pads account IDs, attaches the Bearer token from `localStorage`, and detects MSW vs backend responses (important for offline validation).
  • MUI building blocks: `Card`, `Chip`, `Grid`, `Switch`, `Dialog`, and `LoadingSpinner` deliver consistent spacing + helper text across both screens.
  • Mock infrastructure: MSW handlers (`accountHandlers.ts`, `accountUpdateHandlers.ts`) provide immediate feedback with curated test accounts (`11111111111`, `22222222222`, etc.).

üìä Complexity Guidelines

Simple (1-2 pts) Medium (3-5 pts) Complex (5-8 pts)
  • Simple: Add or restyle a read-only field (e.g., show `eftAccountId` in Account View) using the existing hooks.
  • Medium: Introduce a new validation rule or field dependency (e.g., enforce ZIP + state combos) that requires inline helper text and backend `PUT` coverage.
  • Complex: Integrate an external scoring service or audit log so updates trigger multi-system orchestration (dialog + sequential APIs).

üßæ Data Models

AccountViewResponse (see app/types/account.ts) includes control props (`currentDate`, `transactionId`), account stats (`creditLimit`, `currentBalance`, `groupId`), and customer info (`firstName`, `address`, `customerSsn`).

AccountUpdateData (`app/types/accountUpdate.ts`) contains editable fields such as `activeStatus`, `openDate`, `expirationDate`, `ssn`, and `primaryCardIndicator`.

üìú Business Rules

  • Account IDs must be exactly 11 digits and cannot be all zeroes.
  • Only active accounts (`activeStatus === 'Y'`) display as valid for transactions.
  • Credit available = `creditLimit - currentBalance`; negative balances signal overdraft risk.
  • Masked SSN/card numbers; the toggle persists per-view only (no localStorage).
  • Edit mode + confirmation dialog guard `PUT /accounts/{accountId}` updates.
